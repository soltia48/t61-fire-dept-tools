# フレームスライサ (`t61_frame_slicer`)

スライサの役割は、自走する 2 ビットシンボル列を、整列した 48 バイト
FDMA フレームの列に変換することです。プロトコルの知識は ARIB の同期
ワード 3 種類とフレーム内の正規同期ワード位置のみで、それ以上は持ち
ません。

## 入出力

```
stdin   (4 シンボル/バイト、MSB 先頭) — `--packed-out -` の出力フォーマット
stdout  (1 フレーム 48 バイト)         — `t61_fd_decoder` の入力フォーマット
```

`from_raw_fd(0)` / `from_raw_fd(1)` で記述子を直接開いているため、読み
書きは `read(2)` / `write(2)` システムコールに直結します (Rust 側の
バッファリングなし)。これによりライブ SDR パイプラインのリアルタイム
性が確保されます (
[t61_frame_slicer.rs:21-30](../../src/bin/t61_frame_slicer.rs#L21-L30))。

## フレームレイアウト

| オフセット (シンボル) | 長さ (シンボル) | 内容 |
|---|---|---|
| 0 | 92 | 同期ワード前のデータ |
| 92 | 16 | 同期ウィンドウ (最長同期 SS1) |
| 108 | 84 | 同期ワード後のデータ |
| 192 | — | フレーム終端 |

パックドバイトでは 1 フレーム 48 バイト (`192 / 4`)。

## 同期ワード

ARIB STD-T61 v1.2 part 2 では 3 種類の同期ワードが定義されています。

| 同期 | シンボル列 (MSB ペア先頭) | パック | 用途 |
|---|---|---|---|
| SS1 | `0,2,3,3,2,1,1,0,3,1,0,0,1,2,2,3` | `0x2f94d06b` | センター→端末同期 |
| S2  | `2,1,3,1,0,2,0,3,1,2`             | `0x9d236xxx` | スーパーフレーム先頭 |
| S6  | `0,1,3,2,1,1,1,2,3,3`             | `0x1e56fxxx` | スーパーフレーム途中 |

S2 と S6 は SS1 のウィンドウより短い 10 シンボルです。トップアラインの
32 ビットパターンと 20 ビットマスク (`0xfffff000`) を使うことで、同じ
スライディング比較器で 3 種類すべてに対応できます。
[tables.rs](../../src/tables.rs) と
[slicer.rs:53-69](../../src/slicer.rs#L53-L69) を参照。

## 探索アルゴリズム

```
const SYNC_WORD_OFFSET     = 92    // 期待される同期位置 (シンボル)
const SYNC_WINDOW_SYMBOLS  = 16    // スライディングウィンドウ長
const LP_R_FLUCT           = 4     // 想定ドリフト
const ERROR_MAX            = 1     // 許容シンボルエラー数
```

各候補フレームに対して:

1. シンボルバッファの `SYNC_WORD_OFFSET` 位置から 32 ビットスライディング
   ウィンドウを構築する。
2. **高速路**: 最も可能性の高い位置 (4 シンボル後) を最初に試す。
3. それ以外は `valid - 192` シンボルまで前方に走査し、各ウィンドウを
   `PACKED_SW_TAB` の各エントリのパターン+マスクと比較。
   `(window ^ pattern) & mask` の非ゼロビット数 (Hamming 距離相当) が
   `ERROR_MAX` 以下なら一致。
4. 一致したら、許容内エラーを正規パターンで上書きしてシンボルバッファ
   をクリーニング。

スライサは 3 フレーム分のシンボルバッファを保持し、フレーム送出後は
`FRAME_SYMBOLS - LP_R_FLUCT` (188 シンボル) 分前進させます。これに
より、次回の同期探索で最大 ±4 シンボルのドリフトを吸収できます。

## 無信号処理

現在のスライディングウィンドウで同期が見つからなかった場合、48 バイト
全 0 の "no_signal" プレースホルダ (`NO_SIGNAL_FRAME`) を出力し、
1 フレーム分前進します。下流のデコーダはバイト 0x17 が 0 であることを
見て `"type":"no_signal"` レコードを出力します。これにより、

- 40 ms スーパーフレームの周期が壁時計に合ったまま保たれ、
- デコーダの `frame` カウンタが実時間と一対一に対応する

ことが保証されます。

EOF 時にもバッファに ≥ 10 シンボル残っていれば末尾 no_signal を 1 つ
出力します (フル 1 フレームに満たないが同期一致曖昧性に対応するため)。

## 保留フレーム機構

一致した同期位置が想定ドリフトより遠い場合 (`ret > FRAME_SYMBOLS -
LP_R_FLUCT`)、スライサは取り出した実フレームの *前* に no_signal フ
レームを挿入します。実フレームは 1 スロットの `pending` キューに保存
され、次回の `next_frame` 呼び出しで送出されます。これにより、デコー
ダが期待するフレーム番号付けが完全にズレずに保たれます (1 スロット
失う = no_signal が 1 つ、シフトなし)。
[slicer.rs:236-247](../../src/slicer.rs#L236-L247) を参照。

## 読み出し粒度の契約

探索は欲張り型で、ヒット時かスライド時にしか同期し直しません。その
ため、**入力ソースが短い読み出しを返すと、フル読み出しの場合と異なる
(が、いずれも有効な) フレーム決定**になります。通常ファイル入力で
決定的なフレーミングが必要な場合は、フルチャンクを返す `Read` (例:
`io::stdin()` の `StdinLock` ではなく生ファイル記述子ラッパ) を渡し
てください。ライブパイプ利用では関係ありません — 同期確立後は読み
出し粒度に関わらずフレーミングが一致します。

## ライブラリでの利用

```rust
use t61_fd::Slicer;

let f = std::fs::File::open("symbols.bin")?;
for frame in Slicer::new(f) {
    let frame: [u8; 48] = frame?;
    // ...
}
```

`Slicer` は `Iterator<Item = io::Result<[u8; 48]>>` を実装し、EOF で
fused になります。
